# Обзор проекта diplom_event с учетом листа заданий

## Что уже реализовано в текущем проекте

Текущий код — это базовая платформа афиши на Django с двумя доменами:
- `accounts`: кастомный пользователь и аутентификация;
- `events`: категории, мероприятия, регистрации, календарь, простые рекомендации.

Это хороший фундамент, но он покрывает только часть требований листа заданий.

---

## Сопоставление с листом заданий

### 1) Разработать архитектуру и базу данных

**Что есть сейчас:**
- есть разделение на приложения `accounts` и `events`;
- базовые сущности (`User`, `Category`, `Event`, `Registration`) уже описаны.

**Что добавить:**
- перейти к явной слоям архитектуры: `api/web -> services -> repositories(models)`;
- зафиксировать ER-диаграмму (например, в `docs/architecture.md` + `docs/erd.png`);
- добавить индексы, ограничения и чек-валидации на уровне БД:
  - `end_datetime >= start_datetime`;
  - цена `0` для бесплатных событий;
  - индексы по `status`, `start_datetime`, `creator`;
- вынести конфиг в `.env`, подготовить настройки для production.

### 2) Разработать модуль специалистов

**Что есть сейчас:**
- отдельного модуля специалистов нет;
- уже есть единый пользовательский домен (`accounts`) с ролью в `User`.

**Можно ли без отдельного приложения?**
Да. Формулировку «модуль специалистов» можно интерпретировать как **функциональный модуль внутри общего модуля пользователей**.

**Рекомендуемая реализация (без нового app):**
- оставить один домен пользователей и ввести 2 основные роли: `user` и `specialist`;
- в `accounts` добавить профиль специалиста (`SpecialistProfile`) как `OneToOne` к `User`;
- реализовать в рамках `accounts`:
  - каталог специалистов,
  - карточку специалиста,
  - редактирование профиля специалиста,
  - привязку специалиста к мероприятиям (ведущий/эксперт/организатор);
- добавить фильтры по специализации, локации, рейтингу;
- предусмотреть moderation-поля (`is_verified`, `moderation_status`).

**Когда нужен отдельный `apps/specialists`:**
- если доменная логика специалистов сильно вырастет (сертификация, расписание смен, SLA, биллинг, отдельные права, сложная аналитика).

### 3) Разработать модуль мероприятий

**Что есть сейчас:**
- модуль мероприятий уже существует (CRUD, регистрация, категории, календарь).

**Что улучшить до «production-ready» состояния:**
- заменить изменяющие GET-действия на POST с CSRF;
- добавить валидации бизнес-правил в формы/модели;
- добавить фильтрацию, сортировку, пагинацию;
- разделить роли (организатор/специалист/участник/админ) и доступ;
- добавить статусы жизненного цикла событий и аудит изменений.

### 4) Разработать модуль логики (календарь мероприятий, граф перемещений)

**Что есть сейчас:**
- календарь мероприятий уже есть в базовой версии.

**Что добавить по заданию:**
- выделить логику календаря в отдельный сервис (`CalendarService`);
- добавить модуль графа перемещений:
  - сущности `LocationNode`, `RouteEdge`, `TravelProfile`;
  - расчет маршрута между точками (время/стоимость/длина);
  - подбор оптимальной цепочки посещения мероприятий в пределах окна времени;
- интегрировать карту (например, OSM/Leaflet) и сохранение маршрутов пользователя.

### 5) Разработать модуль отчетности

**Что есть сейчас:**
- полноценного модуля отчетности нет.

**Что нужно сделать:**
- приложение `apps/reports`;
- отчеты по мероприятиям:
  - посещаемость,
  - конверсия записи,
  - отмены,
  - загрузка специалистов,
  - выручка/бесплатные активности;
- экспорт в CSV/XLSX/PDF;
- дашборды для администратора и организатора;
- планировщик периодических отчетов (ежедневно/еженедельно).

### 6) Разработать аналитику с использованием ИИ

**Что есть сейчас:**
- есть простые рекомендации без ИИ.

**Что добавить:**
- приложение `apps/ai_analytics`;
- витрины данных и feature-store (минимально — таблицы агрегатов);
- сценарии ИИ:
  - персональные рекомендации мероприятий,
  - прогноз посещаемости,
  - сегментация пользователей,
  - генерация описаний/анонсов;
- измеримость: метрики качества (CTR/конверсия/MAE), A/B-тесты;
- политики безопасности: логирование промптов, PII-фильтрация, ограничение затрат.

## Приоритизированный roadmap (по этапам)

### Этап 1 (база, 1–2 спринта)
1. Укрепить архитектуру и БД: env-конфиг, индексы, ограничения, миграции.
2. Довести модуль мероприятий до надежного состояния (POST/CSRF, валидации, пагинация).
3. Закрыть тестами критические сценарии.

### Этап 2 (функциональное расширение, 2–3 спринта)
1. Реализовать модуль специалистов (внутри `accounts` или отдельным app — по масштабу домена).
2. Добавить отчетность и админ-дашборды.
3. Вынести календарную логику в сервисный слой.

### Этап 3 (интеллектуальные функции, 2+ спринта)
1. Реализовать граф перемещений и маршрутизацию.
2. Запустить AI-аналитику и персональные рекомендации с метриками.
3. Провести A/B-проверку и итеративно улучшать модели.

---

## Критерии готовности (Definition of Done)

- Для каждого из 6 пунктов листа заданий есть:
  1) выделенный модуль (отдельный app или отдельный функциональный раздел в существующем app),
  2) миграции и модели,
  3) UI/API,
  4) тесты,
  5) документация и метрики.
